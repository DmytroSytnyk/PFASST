3 ## Prerequesites
4 
5 If you use [CMake] to compile, most of the prerequesites are downloaded automatically if they aren't
6 already found on your system.
7 
8 ### Required
9 
10 * A recent C++ compiler that supports (most of) the C++11 standard. This implies either: GCC 4.7
11  (we recommend 4.8 or later) or Clang 3.2 (we recommend 3.3 or later).
12 
13 * The [Boost] `boost::program_options` library (at least version 1.53.0). The [Boost]
14  multiprecision component may also be of interest to \em PFASST++ users.
15 
16 * The [Eigen3] library.
17 
18 ### Optional
19 
20 * The [CMake] build tool (at least version 2.8). This is especially useful for developers.
21 
22 * The [FFTW3] library is required for some of the examples.
23 
24 * The [Google Testing] and [Google Mocking] are required for the unit tests.
25 
26 * An MPI implementation.
27 
28 ### Using [HashDist] to obtain prerequesites
29 
30 The [HashDist] environment management system provides an easy, robust, and version controlled way
31 for our developers (and users) to download and install all prerequesites *locally* (ie, you don't
32 need special privileges on your system to use [HashDist], and it won't modify your system).
33 
34 To use [HashDist]
35 
36 1. Install [HashDist]
37 
38  git clone git@github.com:hashdist/hashdist.git
39  export PATH=$PWD/hashdist/bin:$PATH
40 
41 2. Install prerequesites
42 
43  cd PFASST
44  hit build -v tools/hashdist/pfasst-stack.debian.yaml
45  mv pfasst-stack.debian stack
46 
47 This may take a while (ie, do this before going for lunch) the first time through. However, once
48 complete, subsequent builds (using [CMake] or otherwise) will bypass prerequesite installation, even
49 if you change [CMake] build options or start a fresh out-of-source build directory.

53

58 ## Building with CMake
59 
60 1. Create a directory for out-of-source builds (usually, people are creative and name it `build`) in
61  the source tree (or elsewhere).
62 
63 2. Decide on what and how you want to build:
64 
65  * __General__
66 
67  * By default everything gets built, i.e. tests, examples and all dependencies which are not
68  found on the system.
69 
70  * Use `-DCMAKE_BUILD_TYPE=<VALUE>` to specify general compiler flags.
71 
72  | `CMAKE_BUILD_TYPE` | implied compiler flags |
73  |--------------------|------------------------|
74  | `Debug` | `-g` |
75  | `Release` | `-O3 -DNDEBUG` |
76  | `RelWithDebInfo` | `-O2 -g -DNDEBUG` |
77  | `MinSizeRel` | `-Os -DNDEBUG` |
78 
79  * To enable profiling support, you need to specify `-Dpfasst_WITH_GCC_PROF=ON` and have the GNU
80  compiler selected.
81  When compiling with _Clang_ this option is obsolete as profiling with _Clang_ is not supported.
82 
83  * Users on Linux systems with the _Clang_ compiler and a working installation of LLVM's
84  \em libc++ library may want to activate the usage of that by specifying
85  `-Dpfasst_DISABLE_LIBCXX=OFF`.
86  As libc++ is highly experimental on non-Darwin systems, this is a very exotic option.
87 
88  * __Dependencies__
89 
90  * __Boost__
91  To specify the root path to the Boost installation to be used, add `-DBOOST_ROOT=<PATH>`.
92  The root path should contain the directory `lib` or `lib64` with the compiled Boost libraries
93  as well as `include/boost` with the Boost header files.
94 
95  * __GMock__
96  In case there is the Google Testing and Mocking framework installed on the system in a
97  non-standard path, add `-DGMOCK_ROOT=<PATH>` to specify the root path to the framework.
98 
99  * __MPI__
100 
101  * To enable MPI, please specify `-Dpfasst_WITH_MPI=ON`.
102  To avoid a warning and potential undefined behaviour, also set `-DCMAKE_C_COMPILER` and
103  `-DCMAKE_CXX_COMPILER` to the MPI compiler wrappers.
104 
105  * __Test Suite__
106 
107  * Deactivate building of the test suite by passing `-Dpfasst_BUILD_TESTS=OFF` to the _CMake_
108  command line (not recommended).
109 
110  * __Examples__
111 
112  * Deactivate building of the example programms by passing `-Dpfasst_BUILD_EXAMPLES=OFF` to the
113  _CMake_ command line.
114 
115  * Add compiled example programs to the `install` target by passing `-Dpfasst_INSTALL_EXAMPLES=ON`
116  to the _CMake_ command line (by default, they will not get installed).
117 
118  * __Installing (optional)__
119 
120  * Use `-DCMAKE_INSTALL_PREFIX=<PREFIX>` to specify the prefix for installing the headers and
121  optional compiled examples.
122 
123  By default, `<PREFIX>` is `/usr/local` on Linux systems.
124 
125  The headers will go into `<PREFIX>/include` and compiled binaries in `<PREFIX>/bin`.
126 
127  * In case you also want to install the compiled example programs on your system, specify
128  `-Dpfasst_INSTALL_EXAMPLES=ON`.
129 
130  * For example, to build a release version with _Clang_ on a Linux system without the examples
131  (i.e. only the core test suite and dependencies therefor), the call to _CMake_ looks like:
132 
133  cmake -DCMAKE_BUILD_TYPE=Release -DCMAKE_C_COMPILER=clang \
134  -DCMAKE_CXX_COMPILER=clang++ -Dpfasst_BUILD_EXAMPLES=OFF ..
135 
136  * For a full debug build with enabled profiling on a Linux system with GCC and the desire to
137  install everything to `/home/<USER>`, call _CMake_ with:
138 
139  cmake -DCMAKE_BUILD_TYPE=Debug -Dpfasst_WITH_GCC_PROF=ON \
140  -Dpfasst_INSTALL_EXAMPLES=ON -DCMAKE_INSTALL_PREFIX=$HOME ..
141 
142 3. Compile:
143 
144  make
145 
146 4. Run the unit tests (if unit tests have been built):
147 
148  make test
149 
150  In case any of the tests do not pass, please [open an issue on GitHub][github_new_issue] and
151  provide the full log of your _CMake_ and _make_ invocations.
152 
153 5. (optional) Install \em PFASST++ headers (and compiled examples if specified so):
154 
155  make install
